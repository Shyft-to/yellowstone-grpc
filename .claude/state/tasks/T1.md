## Original task
Create a latency measurement mechanism to measure end-to-end latency from receiving a message to dispatching it to all the subscribers. Put this behind a compile-time flag, and build it as a separate module. We need to plugin this module in all the performance optimization branches or any branch for measuring end to end latency. We will plug it in later.

## Finalized plan

### Sub-task 1: Add `latency-metrics` feature flag to Cargo.toml
Add `latency-metrics = []` to [features] section. No dep: entries.

### Sub-task 2: Create yellowstone-grpc-geyser/src/latency.rs
New module with: `DISPATCH_LATENCY_US` prometheus Histogram (µs buckets 1–10,000), `BATCH_START: Cell<Option<Instant>>` thread-local, `register(&Registry)`, `on_message_received()`, `on_batch_dispatched()`.

### Sub-task 3: Wire lib.rs, metrics.rs, grpc.rs
- lib.rs: `#[cfg(feature = "latency-metrics")] pub mod latency;`
- metrics.rs: `#[cfg(feature = "latency-metrics")] crate::latency::register(&REGISTRY);` in call_once block
- grpc.rs: 4 call sites — 1 receive point (geyser_dispatch Ok(message) arm) + 3 dispatch points (after each CommitmentLevel::Processed broadcast_tx.send)

### Integration
1→2→3 ordering. All changes zero-cost when feature absent.

## Outcome
All 7 evaluation criteria passed: both cargo check variants pass, all 4 instrumentation points correct and cfg-gated, histogram measures receive→Processed-dispatch window on the single geyser_dispatch thread, purely additive (other branches activate with --features latency-metrics).
